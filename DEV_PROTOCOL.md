# 开发协议 — accounting-app

> 本协议由台球小白制定，老王确认后生效。
> 违反协议导致的返工，小白负责。

---

## 原则1：需求冻结再动手

**想法可以随时冒出来，但不写下来不动代码。**

流程：
1. 老王说想法 → 小白整理成需求文档（飞书文档或代码里）
2. 文档包含：要做什么、涉及哪些页面/表、验收标准
3. 老王确认 → 冻结需求 → 开始开发
4. 开发过程中冒出新想法 → 记录到「待办池」，不改当前需求

**例外**：修bug、调UI细节不需要走这个流程。

## 原则2：Demo先行

**先做看得见的东西，再改看不见的东西。**

流程：
1. 先出 UI demo（静态页面或 mock 数据）
2. 老王看 demo → 提反馈 → 调整 demo
3. Demo 确认了 → 接底层代码（数据库、API、业务逻辑）
4. 接完再验证一遍 → 通过 → commit

**为什么**：老王是视觉型思考者，看到画面才知道对不对。写完底层再出页面，改UI等于推翻重来。

## 原则3：一次只做一件事

**不要同时改3个页面。**

- 一个需求一个 commit
- 改完 → 验证 → commit → 再做下一个
- 如果当前改动影响到其他页面，记下来，单独处理

## 原则4：数据库变更走 SQL 文件

**禁止在 Supabase Dashboard 手动加字段/改表。**

- 所有数据库变更写成 `migrations/xxx.sql`
- SQL 文件是唯一的事实来源
- 部署时执行 SQL 文件，不是手动操作

## 原则5：改完立刻验证

每次改动的验证清单：
1. `npx vite build` 构建通过
2. 关键页面手动验证（小白自己先过一遍）
3. `git commit`（不 push）
4. 有测试数据就用测试数据跑一遍

三个都过了才继续。

## 原则6：小白的额外责任

老王是代码小白，只负责提供想法和验收。所以小白要：

1. **预判问题**：老王提一个想法时，主动分析"这个会影响哪些地方"
2. **控制范围**：如果当前需求做到一半老王又提新的，先记录到待办池，不中断当前工作
3. **风险评估**：大改动开始前，先说清楚风险点（哪些地方可能受影响）
4. **定期汇报**：每完成一个步骤告诉老王进度，不要等做完才说
5. **定期复盘**：每隔一段时间主动问"有没有感觉哪里不对劲"

## 原则7：待办池

所有"以后再说"的想法，记录到项目根目录 `BACKLOG.md`：
- 想法描述
- 提出日期
- 优先级（老王定）
- 状态（待做/进行中/已完成）

定期回顾待办池，优先级变了就调整。

## 典型工作流示例

```
老王：我想在电商页面加个退款统计
小白：收到，我先整理需求，出个demo给你看

[小白写需求 → 画demo（mock数据）]

小白：demo好了，你看这个布局对不对
老王：退款要按天显示，不是按月
小白：改好了，再看

[老王确认demo]

小白：demo确认了，我现在接真实数据，大概需要改订单表和视图
老王：好

[小白写SQL migration → 改代码 → 验证 → commit]

小白：做完了，你刷新看看
老王：OK
小白：已commit，等你确认我push上线
```
